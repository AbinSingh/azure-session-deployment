# Layer 1 — Physical Layer (WiFi radio waves)
# Layer 2 — Data Link Layer (The bits become frames.)
# Layer 3 — Network Layer (IP)
# Layer 4 — Transport Layer (TCP)
# Layer 7 — Application Layer


# 1. Is IP part of the OS networking stack? Yes

The networking stack inside the OS typically looks like this:

    Application (redis-cli, Chrome)
            │
    Socket API
            │
    TCP
            │
    IP
            │
    Wi-Fi/Ethernet Driver
            │
    Wi-Fi/Ethernet Hardware (NIC)
            │
    Air / Cable

Everything from Socket API down to the network driver is generally implemented by the operating system kernel.

# 2. Does the router assign the IP address?

Not exactly.

The router usually runs a DHCP server, which offers an IP address.

    Laptop  ------------------> DHCP Discover
    
    Router  ------------------> DHCP Offer
                                 IP = 192.168.1.25
    
    Laptop  ------------------> DHCP Request
    
    Router  ------------------> DHCP ACK

After this exchange, your operating system stores the assigned IP address in memory (kernel networking data structures).

So the IP address is owned by your OS, not by the router.

The router just says: "You may use 192.168.1.25."

# 3. Where is the IP stored?

Inside the OS kernel.

Conceptually something like:

    Wi-Fi Interface (wlan0)
    
    IP Address:
    192.168.1.25
    
    Subnet Mask:
    255.255.255.0
    
    Gateway:
    192.168.1.1
    
    DNS:
    8.8.8.8

These are kernel data structures kept in RAM.

When an application calls: connect(...)

the kernel already knows
    its own IP
    subnet mask
    default gateway
    routing table

# 4. What happens after TCP finishes?
Suppose Chrome wants to send:

GET /

TCP creates something like:

TCP Header
-----------
Src Port: 53421
Dst Port: 80

Payload:
--------
GET /

TCP doesn't know how to reach another machine.

So it hands the segment to IP.

# 5. What does IP do?
The IP layer asks:

"Where is this packet going?"

Suppose the destination is

142.250.190.78

(Google)

IP creates:

    IP Header
    ----------
    Source IP:
    192.168.1.25
    
    Destination IP:
    142.250.190.78
    
    Payload:
    TCP Segment
    
Now IP must decide

"Which network interface should I use?"

It consults the routing table.

Example:

Destination

192.168.1.0/24
↓

Wi-Fi

Everything else

↓

Gateway 192.168.1.1

Since Google isn't on your local network,

IP decides:

Send to Wi-Fi interface.

# 6. Then the Wi-Fi driver
The IP layer passes the packet to the Wi-Fi driver.

The driver is responsible for talking to the Wi-Fi hardware.

It knows things like

how to program the Wi-Fi chip
how to transmit radio frames
interrupts
DMA
hardware registers

The IP layer doesn't know any of this.

# 7. The Wi-Fi driver talks to the hardware

IP Packet

↓

Wi-Fi Driver

↓

Wi-Fi Chip

↓

Radio Waves

The Wi-Fi chip converts the bytes into radio signals.

# 8. Router receives it

The router receives the Wi-Fi frame.

It removes the Wi-Fi header.

Looks at the IP packet.

Reads:

Destination IP

142.250.190.78

Then forwards it toward Google.

# Complete Picture

    Chrome

    ↓
    Socket API

# Summary

The OS already knows its own IP address (obtained earlier via DHCP or configured manually).
When TCP gives data to IP, the IP layer uses the existing source IP address and the destination IP from the socket/connection.
IP creates the IP header and decides how to route the packet (which interface and next hop to use).
