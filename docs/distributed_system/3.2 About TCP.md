# Layer 1 — Physical Layer (WiFi radio waves)
# Layer 2 — Data Link Layer (The bits become frames.)
# Layer 3 — Network Layer (IP)
# Layer 4 — Transport Layer (TCP)
# Layer 7 — Application Layer



Layer 7 application protocol focuses on defining what the bytes mean (RESP), 
while TCP/IP focuses on how the bytes get from one machine to another.



# Let's trace what happens when Chrome or redis-cli calls send().

Your application

Suppose redis-cli has these bytes to send:

*2\r\n
$3\r\n
GET\r\n
$4\r\n
name\r\n

It calls something like:

send(socket_fd, buffer, length);

At this point, redis-cli's job is done.

# The TCP stack (inside the OS)

The kernel's TCP implementation does things like:

Break large data into TCP segments.
Assign sequence numbers.
Track acknowledgments (ACKs).
Retransmit lost packets.
Control congestion.
Buffer data.
Reassemble incoming segments.

For example:

Application gives TCP:

ABCDEFGHIJKLMN...

TCP may split it into:

Segment 1
ABCDEF

Segment 2
GHIJKL

Segment 3
MN...

Each segment gets a TCP header:

+----------------+
| Source Port    |
| Dest Port      |
| Seq Number     |
| ACK Number     |
| Flags          |
+----------------+
| Payload        |
+----------------+
