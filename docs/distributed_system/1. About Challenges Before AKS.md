Before We Move to the AKS Branch

I don't recommend jumping straight to Kubernetes yet.

# There are three important pieces that will make the AKS branch much more meaningful if we cover them first:

1. Replace the in-memory dictionary with Redis on your local machine. 
2. You'll see that almost nothing changes in the application, which demonstrates the value of separating the session store behind create_session(), get_session(), and delete_session().
Containerize the application with Docker. This is how we'll package the app for AKS. 
We'll also observe an important behavior: if the container restarts while using the 
in-memory dictionary, all sessions disappear. Switching to Redis fixes that.
3. Run two FastAPI instances behind a reverse proxy (or simply on different ports). 
With the in-memory dictionary, logging into one instance and sending the next request 
to the other will fail because they don't share session state. Once both instances use 
Redis, the session works regardless of which instance handles the request.

Those three exercises naturally answer the question:

Why do enterprise applications use Redis and Kubernetes instead of just keeping a Python 
dictionary?

By the time we deploy to AKS, you'll understand not only how to deploy it, 
but why each Azure service exists and what problem it solves. 
That's exactly the perspective you'll need when we replace the local session store 
with Azure Cache for Redis and deploy multiple FastAPI replicas on Azure Kubernetes Service.
