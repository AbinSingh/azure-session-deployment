# 1. Big Picture

So far, our session store looks like this:

                     FastAPI Process
                       │
                       ▼
              Python Dictionary (RAM)

    sessions = {
        "abc123": {...},
        "xyz789": {...}
    }

The application and the memory live inside the same Python process.

This creates several problems:
    If FastAPI crashes → sessions disappear.
    If you start a second FastAPI instance → it has its own separate dictionary.
    If the server restarts → everyone is logged out.

Enterprise applications avoid this by moving state outside the application.


### 2. Redis ###

What is Redis?

Many people describe Redis as: "An in-memory key-value database."
Redis is a separate process that specializes in storing and retrieving data extremely quickly over the network.

Before
    Python Process
     ├── API
     └── Session Memory

After
    Python Process
     └── API 
    Redis Process
     └── Session Memory

# Big Picture

Let's first answer a simple question:

What is Redis?
    Redis is not a Python library.
    Redis is not part of FastAPI.
    Redis is a completely independent server application.

Just like:
    Chrome.exe
    Notepad.exe
    python.exe
    redis-server.exe

It's another process running on your operating system.

# 4. What are we installing?
Redis has two main programs.

1. Redis Server

This listens for client connections (default port 6379).

Think of it as similar to:
    Uvicorn listening on port 8000
    SQL Server listening on 1433
    PostgreSQL listening on 5432

2. Redis CLI

This is a client application.
It connects to Redis Server and lets you type commands.

Think of it like:

    Browser  ---> FastAPI
    
    redis-cli ---> Redis

The CLI is our "browser" for Redis.

-- Redis CLI is just a TCP client
-- It uses RESP protocol

# 5.The key idea

redis-cli itself is not a networking layer; it's an application that lives at Layer 7. It uses:
    --RESP as its application-layer protocol.
    --TCP (Layer 4) for reliable transport.
    --IP (Layer 3) for routing.
    --Ethernet/Wi-Fi (Layer 2) for local delivery.
    --The physical medium (Layer 1, and ultimately voltage/light/radio signals) to carry the bits.

Everything below the application layer is provided by the operating system's networking stack and 
the network hardware. 
Redis focuses on defining what the bytes mean (RESP), 
while TCP/IP focuses on how the bytes get from one machine to another.

# 6. Installing Redis on Windows
Option 1 (Recommended for Learning)

Redis doesn't officially support native Windows builds anymore.

The easiest and cleanest approach is to use Memurai Community Edition, which is Redis-compatible and runs natively as a Windows service.

Download it from:

https://www.memurai.com/community

It behaves like Redis and supports the Redis protocol, so the Python client we'll use later works the same way.

# 6.1. Verify the Server is Running
Open PowerShell and run:
    netstat -ano | findstr :6379
Expected output will look similar to:
    TCP    127.0.0.1:6379    0.0.0.0:0    LISTENING    1234

This tells us:
    TCP protocol
    Listening on localhost
    Port 6379
    Process ID (PID) 1234 (your number will differ)

# 6.2. Find the Process
Take the PID from the previous command and run:

    tasklist /FI "PID eq 1234"

Expected output:
    
    Image Name
    -----------
    memurai.exe

Now you've confirmed:

    The OS has created a process.
    The process owns TCP port 6379.
    The process is waiting for clients.

This is exactly analogous to how uvicorn owns port 8000.
