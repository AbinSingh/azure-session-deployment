# But the Browser Doesn't!

Imagine this conversation:

Browser:
Hi, I'm Alice.

Server:
Correct.
Here is your session ID.

Browser:
Thanks.
(Throws it away.)

Next request...

Browser:
Hi.

Server:
Who are you?

That's exactly what is happening.
The browser receives the session ID inside JSON, but it doesn't know it should use it on future requests.
Only cookies (or client-side code) can make the browser send it back automatically.

# Let's Inspect the Session Store

For learning purposes, let's temporarily add a debug endpoint.
Add this to app.py:

from session_store import sessions

@app.get("/debug/sessions")
def debug_sessions():
    return sessions

Now:

Login twice.
Visit:
http://localhost:8000/debug/sessions

You might see:

{
    "c57727b5-4eb3-4df6-bca4-cd8d30764fe4": {
        "username": "alice"
    },
    "d8120a87-bc5d-4f88-95f5-7b1fdbf0b9d3": {
        "username": "bob"
    }
}

This is your server-side session memory.

In production, this dictionary would be Redis.

# A Thought Experiment What happens if you stop Uvicorn?

Visit: /debug/sessions

What do you expect?
Answer:
{}

Why? Because Python dictionaries live only in RAM.

When the process stops, RAM is cleared.

This is one of the biggest reasons enterprise applications use Redis.

Redis keeps the session store outside your application process.
