# Commit 7 — Logout (Destroy the Session)
# Big Picture

Many people think logout simply means Delete the browser cookie.
That is not enough.

# Suppose we only delete the cookie.
    Browser -(No Cookie)
    But the server still has sessions  
    {
        "abc123": {
            "username": "alice"
        }
    }

That session still exists.
Someone who somehow obtained that session ID could continue using it.

# A proper logout has two responsibilities:
    1. Delete the server-side session.
    2. Tell the browser to delete its cookie.
Both must happen.

# Architecture

             Browser
                 │
         POST /logout
                 │
                 ▼
             Middleware
                 │
          request.state.session
                 │
                 ▼
          Logout Endpoint
                 │
        Delete Session Store
                 │
        Delete Cookie Header
                 │
                 ▼
              Browser
         Removes Cookie

Notice something. The middleware still runs first.

The logout endpoint doesn't read cookies directly.

It already has request.state.session_id

# Step 1 — Update session_store.py
    def delete_session(session_id: str):
    sessions.pop(session_id, None)

# Step 2 - Update app.py
    Now add the endpoint:

    @app.post("/logout")
    def logout(request: Request, response: Response):

    if request.state.session_id:
        delete_session(request.state.session_id)

    response.delete_cookie(key="session_id")

    return {
        "message": "Logged out successfully"
    }

Notice how small the endpoint is. Almost all the work has already been done by middleware.

# What does delete_cookie() do?

FastAPI sends a special HTTP response header.

It looks similar to:

    Set-Cookie:
    session_id="";
    Max-Age=0;
    Expires=Thu, 01 Jan 1970 00:00:00 GMT

The browser understands this as
Remove this cookie immediately.

# At the end when executing post/logout
Server remembers nothing.
Browser remembers nothing.

# One Interesting Question
Suppose the browser doesn't delete the cookie.
Imagine it still sends
    Cookie:session_id=abc123

Would login still work? Let's follow the flow.

Middleware:

    Cookie
    ↓
    abc123
    ↓
    Dictionary Lookup
    ↓
    Not Found
Therefore , request.state.session = None

The cookie is now useless because the server no longer has the corresponding session.

This illustrates an important principle:
    The cookie is only a key. The real session lives on the server.

# Before Commit 8

There's one thing we've intentionally ignored so far.

Right now, if Alice logs in today and never logs out, the session stays in memory forever (or until the server restarts).

Enterprise systems don't work that way. They implement session expiration.

We'll enhance the session store so each session has:
    {
        "username": "alice",
        "created_at": "...",
        "last_accessed": "...",
        "expires_at": "..."
    }

Then, on every request, the middleware will determine whether the session is still valid. 
If it has expired, it will remove it and treat the request as unauthenticated.

This will also set the stage perfectly for the AKS branch, because Azure Cache for Redis can automatically 
expire keys using a TTL (time to live), eliminating the need for our application to 
manually clean up expired sessions. That comparison makes it very clear why Redis is such a natural fit for session storage in distributed systems.
