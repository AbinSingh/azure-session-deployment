
# Background

Up until now, our server could answer:
    "Yes, these credentials are valid."

But immediately after that response, it forgot everything.

Now we're going to make the server remember the user.

This is the exact point where a web application changes from stateless to stateful.

# Commit 3 – Generate Session ID and Store Session (No Cookies Yet)

# Big Picture

Our goal in this commit is to introduce server-side state.

We're not sending cookies yet. That comes in Commit 4.

Today's architecture:
                Browser
                    │
        POST /login (username/password)
                    │
                    ▼
               Uvicorn
                    │
                    ▼
                FastAPI
                    │
                    ▼
          Authentication Logic
                    │
            Credentials Valid?
                    │
                    ▼
         Generate Session ID (UUID)
                    │
                    ▼
         Session Store (Dictionary)
                    │
                    ▼
            Return Session ID

Notice something important:

The browser still doesn't automatically remember the session.

We're simply returning the session ID in the JSON response so we can observe it.

Later, we'll replace that with a Set-Cookie header.

# Why separate Commit 3 and Commit 4?

Because there are actually two independent responsibilities:

# Responsibility 1

Server remembers the user.

    Session ID
          │
          ▼
    Dictionary

# Responsibility 2

Browser remembers the Session ID.

    Set-Cookie
          │
          ▼
    Browser Cookie Store

Most tutorials merge these together.

I want you to see they are completely independent.

# What is a Session Store?

Think of it as a Python dictionary.
    {
        "e3ab2...": {
            "username": "alice"
        },
    
        "af72c...": {
            "username": "bob"
        }
    }

Nothing magical.
Just a mapping.

    Session ID
    
    ↓
    
    User Information
