Commit 3 is where the application becomes stateful.

We'll introduce three new concepts together:
    Session IDs — why they must be random and unguessable.
    Session Store — our in-memory dictionary that maps a session ID to user data.
    Cookie Creation — how FastAPI adds a Set-Cookie header so the browser automatically stores the session ID.
This is the point where you'll see the transition from stateless authentication (every request stands alone) to stateful sessions (the server can recognize subsequent requests from the same browser).

Notice something missing. Between Uvicorn and your route, there should be Session Middleware
That middleware is responsible for:
    Reading the Cookie header.
    Extracting session_id.
    Looking up the session dictionary.
    Attaching the session to the request.

Once that's done, every endpoint can simply ask: request.state.session instead of repeatedly parsing cookies.

A small note before Commit 5

The roadmap we discussed earlier had "middleware" as Commit 5, and then "protect /profile" as Commit 6. I suggest we slightly improve that sequence.

In Commit 5, we'll build the middleware and add a temporary /whoami endpoint to inspect what the middleware is doing. That way, before we protect any business endpoint, you'll be able to see:
    the raw Cookie header,
    the extracted session_id,
    the session lookup result,
    and the session object attached to request.state.

Once you can observe those pieces working, protecting /profile in the next commit becomes almost trivial, because the middleware will already be doing the heavy lifting. This mirrors how enterprise applications are developed: first build and verify infrastructure (middleware), then build features (protected endpoints) on top of it.
