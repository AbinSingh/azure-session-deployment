# 1. Big Picture

Imagine Alice logs in. Monday 9:00 AM

Your current dictionary looks like:

    sessions = {
        "abc123": {
            "username": "alice"
        }
    }
Now imagine

    Tuesday , Wednesday, Thursday, Next Month
    That session still exists. Why? Because nothing removes it.

Enterprise systems never allow this.

# 2. Why Expire Sessions?
There are two major reasons.

2.1. Security

Suppose someone steals your session cookie. If sessions never expire,

they can impersonate you forever. Not good.

2.2 Memory

Imagine 10 Million Users

Each session stays forever. Eventually RAM fills up.

# 3. Enterprise Architecture

Today's architecture becomes

    Browser
        │
    Cookie
        │
        ▼
    Middleware
        │
        ▼
    Session Store
        │
        ▼
    Is Session Expired?
        │
     ┌──┴───────┐
     │          │
     ▼          ▼
    Yes         No
     │           │
    Delete       Continue
     │           │
     ▼           ▼
    401        Route Handler

Notice, Middleware becomes smarter.

# 4. Two Types of Expiration

Enterprise applications usually support two kinds.
# 4.1 Absolute Timeout
    Login
    ↓
    30 Minutes
    ↓
    Expired
Even if you're actively using the application.

# 4.2 Idle Timeout
    Login
    ↓
    5 Minutes
    ↓
    Click Something
    ↓
    Timer Reset
    ↓
    Another 5 Minutes
    ↓
    Expired

This is much more common.

We'll implement idle timeout because it's what you'll see in most enterprise applications.

# 5. New Session Structure

Currently:

    {
        "username": "alice"
    }

We'll change it to:

    {
    "username": "alice",
    "created_at": datetime,
    "last_accessed": datetime
    }
